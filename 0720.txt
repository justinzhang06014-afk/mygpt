## 主題四：session 忘記上下文——找到根因並修好了（✅ 已實測驗證）

**根因**（對照容器內實際安裝的 `acp` 套件原始碼確認，`hermes-agent` pip 版本
0.18.2）：`acp_client.py` 的 `run_acp_turn()` 裡，resume_session 分支原本靠
`hasattr(sess, "session_id")` 判斷 resume 有沒有成功——但 ACP 協定的
`ResumeSessionResponse` 這個 model **本來就沒有 `session_id` 欄位**（跟
`NewSessionResponse` 不一樣，resume 時 client 早就知道 session_id 了）。這個
判斷式對「每一次成功的 resume」都會誤判成格式異常，導致**每一輪對話事實上都在
偷偷開一個全新空白 session**，只有 hermes 自己判斷「值得寫進 MEMORY.md/USER.md」
的內容才會留下來，其餘的下一輪就是真的憑空消失——這正是回報的症狀。

**修復**：`acp_client.py` 裡，resume 成功（沒拋例外）就直接沿用呼叫端已經知道的
`resume_session_id` 當 session_id，不再去讀根本不存在的欄位。已修改並重啟容器
生效。

**驗證方式**：3 輪對話測試，turn 1 講一個「不用存記憶」的暗號、turn 2 講無關的話、
turn 3 問暗號是什麼。
- 修復前：log 每輪都印「resume_session 回傳格式異常，改開新 session」，
  hermes 原生對話 ID 每輪都不同。
- 修復後：log 不再出現那行警告，三輪的原生對話 ID 完全一致，暗號正確答對。

這個 bug 從 0716 導入 ACP 之後應該就一直存在，影響的是**所有使用者的每一輪對話**，
不是邊角案例。

## 額外發現：hermes profile create 其實是壞的（silent failure，今天先不修，但已確認影響範圍很小）

原本以為 `hermes profile create` 是真正建立 agent 資料夾的機制。用三種全新路徑
情境實測（完全空白、只先建父層、都不建）——**三次都回報「Profile already exists」
的假錯誤**，即使路徑真的從沒存在過。原因：hermes CLI 啟動時會先在 HERMES_HOME
底下側效應建立 `logs/`，然後同一次呼叫裡的「是否已存在」檢查看到自己剛建的資料夾，
誤判成已存在。

`services.py` 呼叫這個指令的地方（非 clone 分支）完全沒檢查 `returncode`，所以
這個指令其實一直在悄悄失敗，真正建立資料夾/寫 config.yaml/SOUL.md 的是
`_write_isolated_config()`/`_write_soul()` 自己的 Python 程式碼，不是 hermes CLI。

**後續追加驗證，嚴重性比第一眼看起來低**：借用 `--clone-from` 那段已經在用的解法
（HERMES_HOME 指到旁邊暫存資料夾），讓 `hermes profile create` 真的成功一次，
對照它產生的檔案跟我們正式環境的 agent 資料夾差在哪裡。**唯一少的東西是
`profile.yaml`**——查了 `hermes_cli/profiles.py:798-806` 的原始碼註解，寫得很白：
「`config.yaml` 是真正驅動 hermes 的 ~5000 行設定；`profile.yaml` 只是描述這個
profile 本身的裝飾性 metadata（description/role），**缺少這個檔案不會出錯，就是
空預設值**」。也就是說，真正重要的 `config.yaml` 從頭到尾都是我們自己的程式碼在
正確寫，今天所有實測（對話/記憶/MCP/session resume）都成功正是因為這個檔案一直
是對的。結論：**這是死碼+吞掉錯誤的程式碼衛生問題，不是資料遺失或設定失效的
風險**，不急著修，但建議之後找時間把 returncode 檢查補上或乾脆拿掉這段呼叫。

**對 orchestrator 規劃的實際影響**：不需要預先 mount 建好 `profiles/`/`skills/`/
`memories/` 路徑骨架，orchestrator 只要掛一個空的、可寫入的資料夾到 `/opt/data`
就夠了——但也不能指望「hermes 指令自己會建資料夾」，撐起初始化的其實是
hermes-agent 自己的程式碼。這件事今天沒有動手修，只是記錄下來。
