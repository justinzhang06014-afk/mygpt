# 0723 對接文件——四個方向的 POST/GET

三個角色：**使用者端**（別人的前端/後端）、**我的系統**（這支 AI_NEXUS_hermes，192.168.85.82）、**Docker 管理端**（對方 orchestrator，192.168.41.173:5080，以及它建出來的容器本身）。

依照實際跑通的情境A（建立）、情境B（聊天），分成四個方向整理：誰發 POST、誰收/回什麼。

---

## 1️⃣ 使用者端 → 我的系統（使用者端 POST，我的系統回應）

### 1.1 建立/Ensure

```
POST http://192.168.85.82:8643/api/session/ensure
Content-Type: application/json
```

**Request Body：**
```json
{
  "user_id": "demo001",
  "system_prompt": "你是一位專業的 AI 助理，能夠協助使用者處理各種任務。",
  "model": "Qwen/Qwen3.6-35B-A3B-FP8",
  "llm_api_key": "AINX-...",
  "phison_token": "eyJhbGciOi..."
}
```
| 欄位 | 必填 | 說明 |
|---|---|---|
| `user_id` | ✅ | 使用者 ID，會換算成 `agent_id = user_{user_id}` |
| `system_prompt` | 選填 | 留空用系統預設人設 |
| `model` | 選填 | 留空用環境變數 `LLM_MODEL` 預設值 |
| `llm_api_key` | 選填 | 留空用系統共用 key |
| `phison_token` | 選填 | AINexus bearer token，浮動/會過期，建議帶 |

我的系統收到後做的事（見第3、4節）：本地寫好 `config.yaml`/`SOUL.md`/`mcp.json`/`.env` → 呼叫 Docker 管理端建立/確認容器 → 回應下面這包給使用者端。

### 1.2 聊天

```
POST http://192.168.85.82:8643/api/agent/chat/stream
Content-Type: application/json
```

**Request Body：**
```json
{
  "message": "幫我查一下上週的出勤紀錄",
  "user_id": "demo001",
  "room_id": "room_default",
  "phison_token": "eyJhbGciOi..."
}
```
| 欄位 | 必填 | 說明 |
|---|---|---|
| `message` | ✅ | 使用者這輪輸入 |
| `user_id` | 選填但強烈建議帶 | 帶了才會對應到 ensure 時同一個 agent；沒帶就退回 `agent_id`（預設 `agent_default`，多人共用會互相污染） |
| `room_id` | 選填 | 對應 hermes session，預設 `room_default` |
| `system_prompt`／`model`／`llm_api_key` | 選填 | 同 ensure，這輪要換人設/模型才需要帶 |
| `phison_token` | 選填但建議每輪都帶 | 過期/浮動，每輪帶最新的 |

我的系統收到後：先確認/同步遠端容器（見第3、4節），再把回覆用純文字串流吐回去（見第2節）。

## 2️⃣ 我的系統 → 使用者端（我的系統回應／串流給使用者端）

### 2.1 `/api/session/ensure` 的回應

```json
{
  "status": "running",
  "user_id": "demo001",
  "agent_id": "user_demo001",
  "base_url": "http://agent-worker-xxxx:8080",
  "chat_endpoint": "http://192.168.85.82:8643/api/agent/chat/stream",
  "message": "容器建立成功並準備完成",
  "external_base_url": null,
  "worker_id": "a1b2c3d4e5f6"
}
```
**重點：`chat_endpoint` 一律指回我的系統自己**（不是容器的 `base_url`）——使用者端之後每輪聊天永遠打這支固定端點，不用管背後容器是不是換過、有沒有 external proxy，我的系統內部會處理。`base_url`/`worker_id` 只是給你除錯看的，不用打。

### 2.2 `/api/agent/chat/stream` 的回應（純文字串流，`media_type: text/plain`）

正常情況，逐段吐字：
```
你好，幫你查一下...
上週共出勤 5 天...
```

這輪在遠端失敗/被截斷時，會額外附上錯誤說明（不會靜默斷掉）：
```
...原本的部分回覆...
⚠️ 這輪回覆不完整（error）：底層 agent 執行失敗
```

容器目前建不起來/api_server_key 遺失等情況：
```
⚠️ 遠端容器目前沒有可用的認證資訊（api_server_key 遺失），請先重新 ensure 一次。
```

### 2.3 目前沒有、也還不需要的方向：我方主動 push 給使用者

情境A原本有討論「容器建好之後要不要反過去通知使用者」——目前設計是**使用者端呼叫 `/api/session/ensure` 是同步 HTTP request/response**，容器建好與否、位址是什麼，都在這次 response 裡直接給了（2.1），不需要另外設計 webhook/callback 去 push。如果之後使用者端的流程改成「非同步、先發起建立、之後才來查」，才需要另外設計一支 callback 端點（目前尚未實作，也還沒有使用者端的 IP/port 可以填）。

## 3️⃣ 我的系統 → Docker 管理端（我的系統 POST/GET，Docker 管理端回應）

固定打 `http://192.168.41.173:5080`（`ORCHESTRATOR_URL`）。

### 3.1 上傳使用者資料檔案

```
POST /api/v1/users/{user_id}/files
Content-Type: multipart/form-data
```
`user_id` 是字串 `user_id` 用 CRC32 轉換出來的整數。每次會傳這 4 個檔案（建立當下）：`config.yaml`、`SOUL.md`、`mcp.json`、`phison_mcp_bridge.py`；如果有帶 `phison_token`，`.env` 也會一起傳。之後每輪聊天，只會重傳 `.env`（見 3.3）。

**Docker 管理端回應：**
```json
{
  "userId": 3618177923,
  "root": "/home/phison/ainexus/agent-data/3618177923",
  "files": [
    {"path": "config.yaml", "absolutePath": "...", "status": "written", "bytes": 512},
    {"path": "SOUL.md", "absolutePath": "...", "status": "written", "bytes": 1024}
  ]
}
```
我的系統看 `files[].status` 逐筆判斷成功與否（不是看 `writtenCount`，真實回應沒有這個欄位）。

### 3.2 建立容器

```
POST /api/v1/workers
Content-Type: application/json
```
```json
{
  "userId": 3618177923,
  "image": "hermes-agent:latest",
  "environment": {
    "PHISON_API_KEY": "AINX-...",
    "LLM_PROVIDER": "custom",
    "LLM_BASE_URL": "https://ainexus.phison.com/api/external/v1",
    "LLM_MODEL": "Qwen/Qwen3.6-35B-A3B-FP8",
    "LLM_API_KEY": "",
    "API_SERVER_ENABLED": "true",
    "API_SERVER_HOST": "0.0.0.0",
    "API_SERVER_KEY": "<每次隨機生成，只有這一刻拿得到，我方自己存起來>",
    "API_SERVER_CORS_ORIGINS": "*"
  },
  "volumes": {
    "/home/phison/ainexus/agent-data/3618177923": "/opt/data"
  }
}
```

**Docker 管理端回應（201 Created）：**
```json
{
  "id": "a1b2c3d4e5f6",
  "name": "agent-worker-a1b2c3d4e5f6",
  "image": "hermes-agent:latest",
  "userId": 3618177923,
  "status": "running",
  "baseUrl": "http://agent-worker-a1b2c3d4e5f6:8080",
  "externalBaseUrl": null,
  "exitCode": null
}
```
**409 Conflict（該使用者已有容器）：**
```json
{
  "error": "User 3618177923 already has worker 'a1b2c3d4e5f6' for image 'hermes-agent:latest'.",
  "userId": 3618177923,
  "image": "hermes-agent:latest",
  "existingWorkerId": "a1b2c3d4e5f6",
  "reason": "exists"
}
```
撞到 409 時，我的系統會改用下面 3.4 的 GET 拿回這個既有容器的完整資訊。

### 3.3 每輪聊天前：查容器在不在 + 補傳最新 token

```
GET /api/v1/workers
```
回傳整個陣列，我的系統自己過濾 `userId` 找這個使用者的 worker。找得到就只補傳最新 `.env`（走 3.1 同一支端點，只帶一個檔案），不會重新呼叫 3.2。

### 3.4 查詢單一容器

```
GET /api/v1/workers/{worker_id}
```
409 之後、或需要重新確認容器狀態時用。

### 3.5 刪除容器（目前程式有支援，走的是本機 docker.sock 那條路，還沒接到 orchestrator 版本——見下方「已知落差」）

## 4️⃣ Docker 管理端（所建立的容器）→ 我的系統

這裡的「Docker 管理端」指它幫我們建出來的**那個容器本身**（原生 hermes-agent，不是 orchestrator 這支管理服務）。我的系統對它 POST，它用 SSE 串流回應。

```
POST {base_url}/v1/chat/completions
Authorization: Bearer {api_server_key}
X-Hermes-Session-Id: {room_id}
Content-Type: application/json
```
```json
{
  "model": "Qwen/Qwen3.6-35B-A3B-FP8",
  "messages": [
    {"role": "system", "content": "你是一位專業的 AI 助理..."},
    {"role": "user", "content": "幫我查一下上週的出勤紀錄"}
  ],
  "stream": true
}
```

**容器（原生 hermes-agent gateway）的 SSE 回應：**
```
data: {"choices":[{"delta":{"role":"assistant"},"finish_reason":null}]}

data: {"choices":[{"delta":{"content":"上週"},"finish_reason":null}]}

data: {"choices":[{"delta":{"content":"共出勤 5 天"},"finish_reason":null}]}

data: {"choices":[{"delta":{},"finish_reason":"stop"}],"usage":{"prompt_tokens":120,"completion_tokens":45,"total_tokens":165}}

data: [DONE]
```
失敗/截斷時，最後一個 chunk 會是：
```json
{"choices":[{"delta":{},"finish_reason":"error"}],"error":{"message":"..."}}
```
我的系統把這些 `delta.content` 逐段轉成純文字，`finish_reason` 非 `stop` 時把 `error.message` 一併顯示（見 2.2）。

## 已知落差（老實列出來，不是全部都已經打通）

- **`GET /api/users/{id}`、`DELETE /api/users/{id}`（④ 使用者管理）目前只走本機 `runtime_manager.py`（docker.sock）那條路**，orchestrator 模式下呼叫會出錯（實測會回 500，因為內部呼叫的是本機 docker client）。要在 orchestrator 模式下查詢/刪除使用者，目前只能直接打第3節列的 Docker 管理端端點。
- **`HERMES_IMAGE` 的實際名稱**還沒跟 Docker 管理端核對過白名單（`GET /api/v1/agents`），上面範例寫的 `hermes-agent:latest` 是假設值。
- **長駐容器裡的 `phison_mcp_bridge.py`，token 沒辦法在容器活著的期間「即時換」**——3.3 補傳的 `.env` 只在容器第一次開機時有用，之後子行程活著就凍結住舊 token 了（細節見 `0723_light.md` 第5節）。
