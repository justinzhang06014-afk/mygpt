## 專案是什麼

多租戶 AI Agent 平台，讓使用者建立自己的 AI 助理（agent），每個 agent 可以掛 MCP 工具、
安裝技能（skill）、有自己的長期記憶。終極目標是「1000 人連進系統，hermes 幫忙處理各自
機器上的專案」，底層跑的是官方 hermes CLI（Nous Research 的 hermes-agent 套件）。

## 架構流程

Vue(LMFrontend) → chat.js → C#(LlmController.cs) → hermes-agent-proxy(main.py, FastAPI)
→ runtime_manager.py 建立每使用者專屬 Hermes Runtime 容器（複製 hermes-agent-proxy 自己
這個 image，不是官方 hermes-core image）
→ 該容器內用 hermes acp 跟真正的 hermes CLI 溝通（Agent Client Protocol，官方結構化事件
協定，跟 VS Code/Zed/JetBrains 外掛同一套機制，不是猜測文字輸出）

- 每個使用者：`hermes_core_data/users/<user_id>/`，裡面 `profiles/<agent_id>/` 是每個
  agent 自己的資料夾（config.yaml、mcp.json、approvals.json、skills/、memories/、SOUL.md）。
- agent 在 DB 建立當下只寫 Postgres，hermes 自己的 profile 目錄（config.yaml/SOUL.md）
  要等第一次真的呼叫 `ensure_hermes_profile_exists()` 才會建立（第一次對話、或裝
  MCP/技能時順便觸發）——這個時序差是好幾個 bug 的共同根因，見下面。

## 🔴 現在最急的事：git push 被 GitHub 擋下來，因為 commit 裡有真的 API 金鑰

使用者執行 `git push origin main -f` 被 GitHub push protection 擋下：
```
remote: - GITHUB PUSH PROTECTION
remote:   - Anthropic API Key
remote:     commit: 6608d27a8e5ca55a9c4de274b85c184de1570f44
remote:     path: claude api.txt:1
```
已經查證的現況：
- 這個 repo 是**剛 git init 的，只有這一個 commit**（`git log` 只有一筆），且這個 commit
  從沒被 GitHub 接受過（push 被整批拒絕），所以理論上還沒有真的外流到公開的 GitHub 上，
  但金鑰本身已經在本機 git 歷史裡，且透過網路傳輸給 GitHub 掃描過。
- 這個單一 commit 塞了 **3713 個檔案、59MB**，因為**完全沒有 `.gitignore`**，掃到
  `myenv/`（Python 虛擬環境）、一堆 `xxx copy.py` / `123.txt` 之類的暫存檔、
  `hermes_core_data - 複製` 這種備份資料夾都被一起加進去了。
- 除了被 GitHub 標出來的 `claude api.txt`（Anthropic key）之外，**很可能還有其他沒被
  掃到的真實密碼**沒清出來，至少已知：
  - `hermes-agent/config.py:11` 的 `PHISON_LLM_KEY` 明碼預設值
  - `docker-compose.yaml:27` 的 `PHISON_API_KEY`
  - `docker-compose.yaml` 的 Postgres 密碼 `123123`
  - `docker-compose.yaml:97` 的 `ADMIN_TOKEN=change-me-admin-token`（弱預設值本身也算風險）
  - `.env` 檔案（使用者自己開著在 IDE 裡看）

**還沒開始修**。建議的處理方向（沒有動手，等你跟使用者確認）：
1. 先不要再用 `-f` 硬推。
2. 因為只有一個 commit、且從沒被接受過，可以考慮：新增 `.gitignore`（排除 `myenv/`、
   `*.env`、各種 `copy`/暫存檔、`hermes_core_data*`），把 `claude api.txt` 這類含密碼的
   檔案從 tracking 移除，然後用 `git commit --amend` 重寫這唯一一個 commit（不是硬性
   規則裡「禁止 amend」的情境，因為這個 commit 從沒被任何人拉過，等於是本機還沒發布
   的草稿）——但動手前務必跟使用者確認，這是他的 repo 決策。
3. 不管 git 端怎麼清，**這把已經進過 git 歷史的 Anthropic key，建議請使用者當作已外洩、
   直接去 Anthropic Console 註銷重發一把新的**，比在 git 裡清乾淨更重要也更快。
4. 清完之後再重新 push（可能還是需要 force，因為要覆蓋掉本機唯一的舊 commit，但這次
   是正常的「本機 amend 後 push」，跟原本無差別內容硬推不一樣）。

## 測試方法論筆記（省 token/避雷用）

- 使用者對 API 花費很敏感，測試訊息控制在 15 字以內。
- Windows/git-bash 環境下，`curl -d "..."` inline 字串包中文字元常常編碼壞掉
  （C# 端會報 `Cannot transcode invalid UTF-8 JSON text`），一律先寫進暫存 JSON
  檔案再用 `--data-binary @file` 送出。
- `docker exec` 搭配路徑參數（例如 `/app/xxx.py`）在 git-bash 下會被 MSYS 自動轉成
  Windows 路徑而壞掉，指令前面加 `MSYS_NO_PATHCONV=1` 解決。
- 每個 agent 的 Runtime 容器是無狀態的，資料全部在外部 volume（`/opt/data`），容器
  被刪掉/重建不會掉資料，系統有 `/api/runtime/ensure` 自我修復端點會自動重建缺失的
  容器——但**不代表可以隨便刪別人的容器**，任何刪除容器的操作都要先跟使用者確認，
  這條是上一輪真的手滑刪過一次 `hermes-runtime-mm` 換來的教訓。
- 改 `hermes-agent/*.py` 不用重 build image（`./hermes-agent:/app` 是即時掛載），
  但要 `docker restart lm_hermes_agent` 以及對應的使用者 Runtime 容器才會重新載入
  Python 程式碼（uvicorn 沒開 `--reload`）。這個掛載機制連使用者的 Runtime 子容器
  也會複製到，所以只要重啟主 proxy + 該使用者的 runtime container 即可，不用重建 image。
- 改 `LMBackend`（C#）或 `LMFrontend`（Vue）一定要 `docker compose build` 才會生效，
  不是即時掛載。
- 動手改任何程式碼前，先查 hermes 官方原始碼（容器內
  `/usr/local/lib/python3.11/site-packages/hermes_cli/`、`agent/`）核對真實 schema，
  不要憑印象猜 config.yaml 欄位——今天至少有兩次「以前猜錯」被抓出來
  （nudge_interval、skill 需不需要寫回 config.yaml）。
