一、今天修好的 bug（全部都用真實系統重現過，不是憑推論）
--------------------------------------------------

1. 【已修】resume_session 重播事件混進新回覆
   檔案：hermes-agent/acp_client.py
   根因：resume_session() 會把歷史對話透過 session_update 重播一次，這些通知跟這輪真正
   的新事件共用同一個 event_queue，先前完全沒有丟棄機制。
   修法：resume 完、開始新 prompt 前，把佇列裡已經堆積的重播事件全部丟棄。
   驗證方式：程式邏輯修正 + 今晚 Claude API 測試時多輪對話沒有出現舊內容混入。
   殘留風險：如果 hermes 是在 resume_session() 的 RPC 回應「之後」才非同步補送重播通知，
   目前的丟棄時機可能還是會漏接。這個是結構上最合理的修法，但沒有辦法 100% 排除這個
   race condition，之後如果症狀重現要往這個方向查。

2. 【已修】USER.md / MEMORY.md 檔名大小寫
   檔案：hermes-agent/memory_manager.py、hermes-agent/main.py:389（第二個重複的路徑組合處）
   根因：hermes 實際寫出來的檔名是大寫（USER.md / MEMORY.md），但我們兩處地方都組小寫
   檔名。Windows 掛載不分大小寫所以能動，正式 Linux 主機會直接讀不到。
   main.py 那處更嚴重：它是拿小寫路徑做「檔案存不存在」的判斷，判斷錯了會直接回傳假的
   空大腦，連正確的解析函式都不會被呼叫到。
   驗證方式：實際 ls 過 hermes_core_data 底下的真實檔案，確認 hermes 真的輸出大寫。

七、Agent 分享功能 —— 前瞻性設計（你要求先想清楚、不要現在動手做）
--------------------------------------------------------------

【背景結論，已查證 hermes 原始碼/CLI】
hermes 原生就有完整的「打包/分享/安裝」機制，不需要我們自己刻：

    hermes profile export <name>      打包成 .tar.gz
    hermes profile install <來源>      來源可以是 git URL（github.com/user/repo、
                                       https://...、git@...）或本機資料夾，資料夾
                                       根目錄要有 distribution.yaml 這份manifest
    hermes profile update             重新拉取來源、套用更新，且不會洗掉安裝方
                                       自己的個人化資料（記憶、對話紀錄）

【你在動手前應該先讀哪些資訊】
1. distribution.yaml 的完整 schema——目前我只查到指令存在、參數是什麼，還沒有
   實際 export 一次去看 tar.gz 裡面實際包含哪些檔案、manifest 長什麼樣子。這是
   動手做之前的第一步，直接 hermes profile export 一個現有 agent 看真實輸出。
2. export 出來的內容範圍——需要驗證是否包含 memories/skills/mcp_servers 三者全部，
   還是預設只包一部分（例如 SOUL.md + config.yaml，skills/memories 可能要另外參數）。
3. 憑證安不安全——已經確認一個好消息：我們自己 mcp_services.py 寫進 config.yaml 的
   MCP 憑證本來就是用 ${環境變數} 引用格式，不是明碼，所以理論上匯出分享不會連密碼
   一起外洩。但這只是「我們自己寫的格式」是安全的，還沒驗證 hermes profile export
   打包時會不會額外把 .env 檔案也包進去（如果連 .env 一起包，安全假設就破功了，
   這件事沒查過，是動手前必須先驗證的）。

【我會怎麼設計】

核心原則：不要重新發明，讓 hermes profile export/install 做真正的搬運工作，我們的
角色只是「在 hermes 那組指令外面包一層權限/選擇性/使用者體驗」。

分成兩個角色：
- 發布方（Publisher）：想把自己調教好的 agent 分享出去的人
- 安裝方（Installer）：想直接拿別人做好的 agent 來用、省去重新訓練時間的人

發布方的流程設計：
1. 在 agent 卡片上加一個「📤 發布分享」按鈕（跟現有「匯出」功能同一個選單，多一個選項）
2. 跳出一個「選擇性分享」畫面（回應你問的「假設使用者只願意分享一部份」）：
   - MCP 清單勾選：這個 agent 目前裝的每一個 MCP，逐一勾選「要不要包進分享包」。
     不勾的，分享包裡就不會出現這個 MCP 的宣告（config.yaml 的 mcp_servers 區塊
     那筆整個拿掉，不是留著但停用，避免安裝方誤以為有這個能力）。
   - Skills 清單勾選：同理，逐一勾選要分享哪些技能資料夾（不勾的技能資料夾在打包
     前先排除，不要整個 skills/ 目錄照搬）。
   - 明確警語：MCP 的憑證（.env 裡的值）一律不會被包進分享包，安裝方拿到的是
     「需要自己填憑證的 MCP 宣告」，不是現成能用的憑證。
3. 後端實作：不要直接呼叫 hermes profile export 對整個 profile 打包（那樣沒辦法
   選擇性排除），改成：
   a. 複製一份該 agent 的 profile 到一個暫存目錄
   b. 依照使用者勾選，從這份暫存複製體裡刪除沒勾選的 MCP 宣告（改 config.yaml）
      跟沒勾選的技能資料夾
   c. 強制刪除暫存複製體裡的 .env（憑證來源）
   d. 對這個「已經過篩選的暫存目錄」執行 hermes profile export
   e. 把產出的 .tar.gz 提供下載，或者更進階：推到一個我們自己管理的
      「agent 分享商店」（類似現在 MCP/skill 的母版模式，但存的是整包 tar.gz
      的下載連結 + 說明文字 + 這個分享包實際包含哪些 MCP/skill 的清單，讓安裝方
      在下載前就能看到「這包裡有什麼」，不用下載完才發現不是他要的）

安裝方的流程設計：
1. 建立 agent 彈窗（我們今晚剛做的那個）增加第三個入口：「從分享包建立」
2. 選一個分享商店裡的項目 → 後端呼叫 hermes profile install <tar.gz 或 git 位置>
   --name <使用者自訂的新名稱>
3. 安裝完成後，畫面明確列出「這個分享包包含哪些 MCP（需要你自己填憑證才能用）」，
   引導使用者直接跳轉到 MCP 商店把憑證欄位補上，而不是讓使用者裝完才發現東西
   不能用、自己要找半天

【使用者情境（具體例子）】
情境：小美訓練了一個很強的「財報分析師」agent，裝了 GitLab MCP（連公司內部 GitLab，
有敏感 token）跟 Email MCP（她自己的私人信箱），也裝了 excel-author 技能。
她想分享給同事小華，但不想把公司 GitLab token 或自己的信箱密碼給出去，
也不覺得小華需要 Email 這個功能。

用上面的設計，小美在「發布分享」畫面只勾選 excel-author 技能，MCP 兩個都不勾。
產出的分享包只有 SOUL.md（人設）+ excel-author 技能，沒有任何 MCP 宣告、沒有
任何憑證。小華安裝後拿到一個「會做 Excel 報表的財報分析師人格」，但沒有小美的
GitLab/Email 存取權——如果小華自己也需要 GitLab MCP，他要自己去 MCP 商店重新掛，
